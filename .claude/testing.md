# Философия и правила тестирования

## Принцип

Проект следует **стартап-ориентированной философии тестирования**: тестируем только то, что приносит максимальную ценность при минимальных затратах.

Единственная область с обязательными тестами — **детерминированная бизнес-логика**.

---

## Что ОБЯЗАТЕЛЬНО тестировать

| Область | Примеры |
|---------|---------|
| Игровой движок (`core/engine`) | `generateRowHints`, `generateColumnHints`, `applyMove`, `checkSolved` |
| Доменные алгоритмы | любая новая чистая функция в `core/engine` |
| Критическая бизнес-логика | валидация данных, вычисление подсказок |

**Правило:** любой новый алгоритм в `core/` должен сопровождаться unit-тестами.

---

## Что НЕ нужно тестировать

| Область | Причина |
|---------|---------|
| UI-виджеты | Слишком хрупко, высокая стоимость поддержки |
| Навигация (GoRouter) | Фреймворк протестирован авторами |
| Провайдеры Riverpod | Зависят от контекста Flutter |
| Presentation-логика | Покрывается ручным QA |
| Инициализация БД | Drift тестируется авторами |

## Структура тестов

Каждая тестируемая функция — отдельный файл. Общие хелперы вынесены в `puzzle_test_helpers.dart`.

```
test/
  core/
    engine/
      puzzle_test_helpers.dart        ← buildPuzzle(), buildState()
      generate_row_hints_test.dart    ← PuzzleEngine.generateRowHints
      generate_column_hints_test.dart ← PuzzleEngine.generateColumnHints
      apply_move_test.dart            ← PuzzleEngine.applyMove
      check_solved_test.dart          ← PuzzleEngine.checkSolved
```

**Правило:** один тестовый файл = одна тестируемая функция.

## Консольный вывод и CI/CD

В корне проекта находится файл `dart_test.yaml`:

```yaml
reporter: expanded
```

Это делает развёрнутый вывод поведением по умолчанию. При запуске `flutter test` каждый тест отображается отдельной строкой — имена тестов служат живой документацией поведения.

**Пример вывода:**
```
PuzzleEngine.generateRowHints
  ✓ паттерн «крест» 5×5: каждая строка даёт верную подсказку
  ✓ строка полностью пустая (все нули) → подсказка [0]
  ✓ несколько групп в строке: [1,1,0,1,0,1,1,1] → подсказка [2, 1, 3]

PuzzleEngine.applyMove
  ✓ fill режим: тап по пустой ячейке закрашивает её (empty → filled)
  ✓ cross режим: повторный тап по ячейке с крестом убирает крест (cross → empty)
```

**В CI/CD** тот же `flutter test` возвращает exit code 0 при успехе и 1 при падении. Развёрнутый формат читается в логах GitHub Actions, GitLab CI и любых других.

## Запуск тестов

### Предварительное условие

Тесты используют `@freezed`-сущности, которые требуют кодогенерации:

```bash
flutter pub run build_runner build --delete-conflicting-outputs
```

### Запуск всех тестов

```bash
flutter test
```

### Запуск конкретного файла

```bash
flutter test test/core/engine/apply_move_test.dart
```

## Что тестируется сейчас

21 тест в 4 файлах:

**`generate_row_hints_test.dart`** (6 тестов)
- Паттерн «крест» 5×5
- Пустая строка → `[0]`
- Полностью заполненная строка
- Несколько групп
- Пустой `solution`
- Одна ячейка

**`generate_column_hints_test.dart`** (5 тестов)
- Паттерн «крест» 5×5
- Пустой столбец
- Полностью заполненный столбец
- Несколько групп по вертикали
- Пустой `solution`

**`apply_move_test.dart`** (5 тестов)
- Fill: empty → filled
- Fill: filled → empty (toggle)
- Cross: empty → cross
- Cross: cross → empty (toggle)
- Только целевая ячейка изменяется

**`check_solved_test.dart`** (5 тестов)
- Совпадение с решением → `true`
- Пустая сетка → `false`
- Одна лишняя ячейка → `false`
- Кресты на пустых не мешают победе
- Полный 5×5 паттерн «крест»

## Инструменты

Используем только стандартный `flutter_test` из SDK Flutter.

Не добавлять: Mockito, mocktail, golden_toolkit или другие внешние тестовые фреймворки без явной необходимости.
