# Архитектурные решения (ADR)

## ADR-001: Feature-Sliced Architecture

**Дата:** 2026-03-16

**Решение:** Использовать Feature-Sliced Architecture, адаптированную для Flutter.

**Причина:** Позволяет добавлять новые фичи изолированно, не затрагивая существующий код. Каждая фича — вертикальный срез от UI до репозитория.

**Альтернативы:** Layered (горизонтальные слои), Clean Architecture.

## ADR-002: Riverpod вместо Bloc

**Дата:** 2026-03-16

**Решение:** `flutter_riverpod` для управления состоянием.

**Причина:** Меньше boilerplate по сравнению с Bloc. Compile-safe провайдеры. `FutureProvider.family` удобен для загрузки уровней по параметру сложности.

## ADR-003: Подсказки генерируются в runtime

**Дата:** 2026-03-16

**Решение:** `rowHints` и `colHints` не хранятся в JSON или БД. Вычисляются из `solution` при загрузке через `PuzzleEngine`.

**Причина:** Единственный источник правды — `solution`. Хранение подсказок создаёт риск рассинхронизации. Вычисление тривиально и не требует ресурсов.

---

## ADR-004: Передача Puzzle через StateProvider

**Дата:** 2026-03-16

**Решение:** Для передачи пазла на экран игры используется `selectedPuzzleProvider = StateProvider<Puzzle?>`.

**Причина:** Пазл содержит `List<List<int>>`, которые неудобно сериализовать в URL-параметры GoRouter. `StateProvider` даёт типобезопасный способ передачи объекта между экранами.

**Ограничение:** При глубоком линке (deep link) на экран игры нужно будет пересмотреть эту схему.

## ADR-005: SQLite только для прогресса

**Дата:** 2026-03-16

**Решение:** В БД хранится только таблица `progress_entries`. Уровни — в JSON-файлах в assets.

**Причина:** Уровни статичны и поставляются с приложением. Нет смысла хранить их в БД. Прогресс динамический — нужна персистентность.

---

## ADR-006: `appDatabaseProvider` определён в `core`, не в `features`

**Дата:** 2026-03-16

**Решение:** Провайдер базы данных `appDatabaseProvider` живёт в `lib/core/database/database_provider.dart`.

**Причина:** `AppDatabase` — инфраструктурный компонент слоя `core`. Если провайдер определён внутри `features/progress/`, то любая другая фича, которой нужна БД, вынуждена импортировать из чужой фичи — это нарушает архитектурное правило изоляции фич. Размещение в `core` даёт корректный доступ всем фичам без нарушения зависимостей.
