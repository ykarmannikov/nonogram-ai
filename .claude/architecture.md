# Архитектура проекта

Проект использует **Feature-Sliced Architecture**, адаптированную для Flutter.
Код разбит на пять слоёв с однонаправленными зависимостями: `shared` ← `entities` ← `core` ← `features` ← `app`.

---

## Слой `app`

**Директория:** `lib/app/`

Точка входа и корень приложения. Содержит `app.dart` (корневой виджет, тема) и `router.dart` (GoRouter-конфигурация). Слой не содержит бизнес-логики — только bootstrap: обёртка `ProviderScope`, привязка темы и маршрутов. При расширении сюда добавляют глобальные `Override`-ы провайдеров для тестирования или feature-флаги.

---

## Слой `core`

**Директория:** `lib/core/`

Инфраструктура и движок игры. Разделён на три модуля:

- **`core/database/`** — конфигурация Drift-базы данных, определение таблиц и DAO-методов. Единственное место, где происходит прямое взаимодействие с SQLite.
- **`core/engine/`** — чистые функции игровой логики (`PuzzleEngine`): генерация подсказок из решения, применение ходов, проверка победы. Не имеет зависимостей от Flutter или провайдеров.
- **`core/utils/`** — вспомогательные утилиты (загрузка JSON из assets).

Ничего из `core` не знает о `features` или `app`.

---

## Слой `entities`

**Директория:** `lib/entities/`

Доменные модели данных. Содержит только иммутабельные структуры данных (`@freezed`) и перечисления. Не содержит логики, зависимостей от Flutter или Riverpod. Это наиболее стабильный слой — модели меняются редко.

Модели: `Puzzle`, `GameState`, `Progress`, `CellState`, `GameMode`.

---

## Слой `features`

**Директория:** `lib/features/`

Пользовательские сценарии. Каждая фича — самодостаточный вертикальный срез:

```
features/
  difficulty_select/   — выбор сложности
  level_select/        — список уровней
  game/                — игровая сессия
  progress/            — управление прогрессом
```

Внутри фичи допустима структура `ui/`, `state/`, `repository/`. Фичи могут использовать `entities` и `core`, но **не должны** напрямую импортировать друг друга. Межфичевое взаимодействие через провайдеры Riverpod (например, `selectedPuzzleProvider`).

### Unlock-логика уровней

Правила разблокировки реализованы в `features/level_select/state/levels_provider.dart`:

- **Easy-уровни**: всегда разблокированы.
- **Hard-режим**: кнопка на `DifficultyScreen` заблокирована, пока не пройдены все easy. Контролируется `hardUnlockedProvider` — смотрит `levelsProvider('easy')` + `progressProvider`.
- **Hard-уровни внутри списка**: последовательная разблокировка — `hard_N` доступен только после прохождения `hard_{N-1}`. Логика в `LevelSelectScreen._isLocked()`.

Поток данных: `progressProvider` (БД) → `hardUnlockedProvider` → `DifficultyScreen` (кнопка) и `LevelSelectScreen` (карточки).

## Слой `shared`

**Директория:** `lib/shared/`

Переиспользуемые UI-компоненты и тема. Содержит только «немые» виджеты (stateless, без бизнес-логики) и конфигурацию темы. Не знает ни о каких фичах или доменных моделях. Если компонент требует доступа к провайдерам — он принадлежит к конкретной фиче, а не к `shared`.

---

## Направление зависимостей

```
app → features → core → entities
            ↘         ↗
              shared
```

Нарушение этой схемы — антипаттерн (см. `anti_patterns.md`).
