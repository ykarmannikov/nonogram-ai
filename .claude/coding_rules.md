# Правила разработки

## 1. Никакой бизнес-логики в UI

Виджеты только отображают данные и вызывают методы нотификаторов.

**Нельзя:**
```dart
// В виджете
final solved = grid.every((row) => row.every((cell) => ...));
```

**Нужно:**
```dart
// В PuzzleEngine или GameNotifier
final solved = PuzzleEngine.checkSolved(state);
```

## 2. Иммутабельные модели

Все доменные модели — `@freezed`. Изменение состояния только через `copyWith`.

**Нельзя:** мутировать `playerGrid` напрямую — создавать копию через `List.from`.

## 3. Производные состояния не хранятся

Не хранить в БД то, что можно вычислить:
- Подсказки → вычислить из `solution`
- Разблокованные уровни → вычислить из `progress`
- Количество ходов → не нужно в MVP

## 4. Слои не нарушают зависимости

- `shared` не импортирует `features` или `core`
- `entities` не импортирует `flutter_riverpod`
- `core/engine` не импортирует `features`

## 5. Каждая фича отвечает за свои данные

Фича `game` не должна напрямую обращаться к репозиторию `progress`. Используется `progressProvider` из фичи `progress`.

## 6. Один нотификатор — один сценарий

`GameNotifier` — активная партия. `ProgressNotifier` — список пройденных уровней. Смешивать нельзя.

## 7. Новые уровни через JSON

Добавление уровней — только создание нового JSON-файла в `assets/puzzles/` и регистрация пути в `LevelsRepository`. Без изменения кода.

## 8. Комментарии только там, где неочевидно

Тривиальный код не комментировать. Комментировать: нетривиальные алгоритмы, архитектурные решения, причины ограничений.

## 9. Новые алгоритмы в `core/` требуют тестов

Каждая новая детерминированная функция в `core/engine/` или `core/utils/` должна быть покрыта unit-тестами в `test/`.

Тесты НЕ нужны для: UI, виджетов, провайдеров, навигации.

Подробнее — в `testing.md`.

## 10. Lint-правила

Проект использует `package:flutter_lints/flutter.yaml` как базу + дополнительные правила в `analysis_options.yaml`.

**Активные правила:**

| Правило | Что проверяет |
|---------|---------------|
| `prefer_const_constructors` | Используй `const` там, где возможно |
| `prefer_const_literals_to_create_immutables` | `const [...]` вместо `[...]` для неизменяемых литералов |
| `prefer_final_fields` | Поля, которые не переприсваиваются — `final` |
| `prefer_final_locals` | Локальные переменные, не переприсваиваемые — `final` |
| `always_declare_return_types` | Всегда явно указывай тип возврата (в т.ч. `Future<void>` для async-методов) |
| `avoid_print` | Не использовать `print()` в продакшн-коде |
| `avoid_unused_constructor_parameters` | Параметры конструктора, которые не используются |
| `unnecessary_this` | Убирай `this.` там, где оно избыточно |
| `sort_constructors_first` | Конструкторы — первыми в классе |
| `sort_child_properties_last` | `child`/`children` — последними в виджете |

**Важно:** правило `avoid_unused_parameters` не существует в Dart. Используй `avoid_unused_constructor_parameters`.

**Генерируемые файлы исключены** (`*.g.dart`, `*.freezed.dart`, `build/**`) — ошибки в них не блокируют сборку.

Подробнее о CI — в `ci.md`.
